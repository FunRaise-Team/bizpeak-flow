# BizPeak Flow — CONSTITUTION（憲法草案 v0.1）

> 本檔是 BizPeak Flow 的不可退讓原則、半年不變等級。動這個 repo、寫 SPEC、開卡帶、派代理之前**先讀這頁、引用它、不重新協商**。修憲需 Nelsen 拍板。
> 方法論：GitHub Spec Kit（constitution → spec → plan → tasks）。
> **狀態**：草案、待 Nelsen 批准後生效。

---

## 第一條：Notion 是唯一真相來源

所有業務資料（客戶、報價單、合約、款項、催收記錄）的 canonical store 是 **Notion Database**。

- 任何介面（1.0 GUI / 1.5 Agent / 2.0 MCP）與任何卡帶、都是 Notion 之上的**視圖與流程**
- 允許快取、允許索引、**不允許第二真相來源** — 誰要另建資料庫存業務資料、先修憲
- Why：Nelsen 明定「所有資料盡量都能回到 Notion」；團隊已在 Notion 工作、資料不落地到大家看不到的地方才能解「離職 = 知識蒸發」的風險

## 第二條：能力先於介面（API-first）

任何功能先做成**能力層（可程式呼叫的動作與查詢）**、再包 UI。

- 一個功能做完的定義：GUI 按鈕按得到、Agent 講得動、MCP 問得到 — 三種介面吃同一個能力層
- UI 是皮、能力是骨。先長骨、再貼皮
- Why：這是 1.0 → 1.5 → 2.0 三階段體驗能夠演進而不重寫的唯一路徑

## 第三條：卡帶合規、主幹道不分岔

平台（主機）與功能模組（卡帶）嚴格分離。

- 新功能一律以**卡帶**形式接入、必須通過 `docs/CARTRIDGE-SPEC.md` 的接入契約（manifest、資料契約、能力層、事件）
- 不合規的點狀工具**不得**接入主幹道 — 可以先當 L0 私人卡帶養、要升級就補合規
- 平台層改動需 Decision Owner 拍板；卡帶層各自演化、互不阻塞
- Why：AI 先鋒計畫的 Vibe Coding 能量要導進軌道、不是壓熄 — 過去點狀工具無法串聯的教訓不重演

## 第四條：關鍵節點必有人

簽核、用印、開發票、收款確認、對外寄信 — 法律與財務動作**必須有人類確認**。

- 系統的職責是：準備好一切、追蹤到底、提醒到人 — 不是替人蓋章
- 自動化程度只能漸進上調（手動 → 半自動 → 全自動）、每次上調需流程 owner 同意
- Why：B2B 合約金額大、錯一次的代價遠高於省下的點擊

## 第五條：每個狀態變化都留痕

合約生命週期的每次狀態轉移都要有 audit trail：誰、何時、什麼動作、憑什麼（信件/檔案連結）。

- 落檔不是選配、是這個系統存在的理由之一（教育訓練成本、離職風險）
- 完成宣告必附證據 — 系統層面的 Verify Before Claim

## 第六條：對內對外預設繁體中文

介面文案、通知信件、文件預設繁中；技術名詞與程式碼保留英文。

## 第七條：內部工具安全底線

部署到任何可公開存取的環境、預設鎖公司 IP 或走 SSO；禁 client-side password 門。合約與財務資料是公司最敏感資料之一。
