## 1. 一句話定位與品牌座標

**從報價單產生的那一刻、到收款與續約的完整閉環** — FUNRAISE 內部合約生命週期系統、同時是 BizPeak 生態系第一個 Dogfooding 產品、以及公司工作流「主幹道」的第一段路。

**品牌座標**（依公司品牌架構頁、Microsoft Teams 命名模式）：
- 方睿科技 FUNRAISE（母品牌）→ **生態系事業群「釋放專業」→ BizPeak 子品牌** → 產品 = BizPeak + 使用情境
- **BizPeak Flow** 與 BizPeak Call / BizPeak Chat / BizPeak Studio（名稱 TBC）平行、共享品牌記憶
- 品牌圖已有「Funraise MCP — 橫跨兩群的共用資料接口」：本產品的體驗 2.0 能力層就是它的生態系資料拼圖之一、不另造輪子

## 2. 目標與非目標

### 目標（v1）
1. 合約簽核流程全程可追蹤 — 台北客戶「發票寄了、款沒收、月底才發現」不再發生
2. 流程知識落檔 — 新人 1 小時上手、離職不帶走流程
3. 收款與續約的主動通知 — 系統追人、不是人追表
4. 建立卡帶規範 — AI 先鋒計畫的產出有軌道可接

### 非目標（v1、明確排除）
- 不重建 CRM / 業務漏斗（AI-raiser-planning 儀表板已管 lead → contract 七階段、我們從「確認合作」接手）
- 不自建「對外數位簽章」的憑證引擎（CA 憑證、法律推定效力那層外接 DottedSign 等、研究結論：連 Ironclad 都外接）— **內部簽核引擎則自建、見 §7**
- 不做對外多租戶 SaaS 版（架構預留、商業模式本輪不展開 — 會議記錄第⑧節）

## 3. 使用者與三入口

| 入口 | 使用者 | 核心畫面 |
|------|--------|----------|
| **個人入口** | 業務（每位同仁） | 我的合約、我卡在哪一步、我的催收任務 |
| **行政・財務入口** | Ting（總務）/ Glendy（會計） | 用印佇列、開票佇列、收款打勾、逾期款項表 |
| **Funraise 入口** | Nelsen / BU 主管（Philis） | 現金流量預測（預計付款日 × 金額）、全合約看板、逾期 KPI |

## 4. 合約生命週期狀態機

融合 Philis 九步流程與 Allen 架構底稿的 10 格狀態機（單向前進、退回留痕、樂觀鎖、狀態定義集中單一檔案）：

```
C0 報價草稿 ──> C1 內部簽核中 ──> C2 用印完成 ──> C3 已寄出待回簽 ──> C4 已回簽生效
（接報價單系統）   （自建簽核引擎）   （待寄出）      （SLA + 自動提醒）   （自動通知財務）
                                                                        │
        ┌───────────────────────────────────────────────────────────────┘
        v
C5 履約・收款中 ──> C6 結案 ──> C7 續約窗口（到期前 60/30 天觸發 → 回 C0 開新約）
（款項子表驅動）
終端分支：X1 作廢（留痕）/ X2 不需寄出歸檔
```

每格狀態帶：SLA 天數、負責人、進入時間、憑證連結（信件 / 檔案）。所有轉移寫入事件記錄（憲法第五條）。

### 款項子狀態機（一約多期、CT-N 子表掛在合約下）

```
P0 排程 ──> P1 已開票 ──> P2 已通知客戶 ──> P3 已收款（財務打勾）
                                  │
                                  └──> P4 逾期 ──> 催收梯級
```

**催收梯級（規則引擎、可依客戶分群配置）**：
- D+3：提醒負責業務（個人入口 + 通知）
- D+7：草擬催收信給業務確認後寄出（憲法第四條：對外信件人審）
- D+30：升級 BU 主管、進逾期款項表
- D+60〜90：強制「升級決策」節點（正式催告 / 停止服務 / 委外）— 研究依據：逾期 90 天回收率降至約五成

## 5. 資料模型（Notion 資料層、憲法第一條）

| Database | 關鍵欄位 | 備註 |
|----------|----------|------|
| **Contracts 合約主檔** | 客戶（relation → 既有客戶大名單 DB）、狀態（C0-C7/X1-X2）、負責業務、事業群、金額、生效日 / 到期日、回簽檔案、SLA 截止 | 新建；與儀表板共用同一客戶權威源 |
| **Payments 款項** | 合約（relation）、期數、預計付款日、金額、發票號、狀態（P0-P4）、實收日 | rollup 到合約主檔彙總未收 / 逾期 — 直接餵現金流量表 |
| **Events 事件記錄** | 誰、何時、什麼動作、憑證連結 | audit trail、不可刪改 |
| **Cartridges 卡帶註冊** | 卡帶名、owner、等級（L0-L2）、資料契約、暴露能力 | 主幹道治理（詳 CARTRIDGE-SPEC） |

規模驗證：Notion 付費版 20,000 rows/DB、效能甜蜜點 5,000 rows 內 — 以公司 B2B 合約量級（年百張內）撐 10 年以上。快取層擋 3 req/s 限流（詳 ARCHITECTURE）。

## 6. 通知矩陣（系統追人）

| 事件 | 受眾 | 時機 |
|------|------|------|
| 客戶回簽收到（sales@ 進信） | 財務 Glendy | 即時、自動 |
| 開票後到期未收款 | 負責業務 + 財務 | 催收梯級 |
| 合約到期前 60 / 30 天 | 負責業務 + BU 主管 | 續約窗口 |
| 簽核 / 用印卡關超過 SLA | 該狀態負責人 + 發起人 | 每日檢查 |
| 現金流量週報（預計付款彙總） | Nelsen / 管理層 | 每週一 |

通道：Teams（既有協作場）為主、Email 為輔；通道優先序待訪談確認。**失敗告警原則**：任何整合失敗必須出聲、不得靜默（Allen 底稿明列的缺口、引為前車之鑑）。

## 7. 電子簽核與簽署（自立為主、分級設計）

**方向修正（2026-07-02 Nelsen 拍板）：內部簽核引擎自建、不依賴 NuEIP。** 簽核是主幹道的核心能力（未來每個卡帶的流程都要用）、外包給人資系統等於把主幹道的心臟放在別人身上。NuEIP 降為**可選接口** — 個別流程若公司政策要求回接（如人事類）、走 adapter 接上、但大部分流程自立完成。

**自建簽核引擎（平台服務、所有卡帶共用）**：
- 簽核鏈定義：依合約金額 / 事業群自動路由（誰簽、順序、平行或串行）、鏈定義是資料不是程式碼
- 代理人機制：簽核人請假 → 自動轉代理、不卡流程
- 行動端批准：Teams 卡片上直接核准 / 退回（附意見）、不用開系統
- 沿途留痕：每一步誰簽、何時、意見、憑證 — 進 Events（憲法第五條）
- 用印整合：簽核完成 → 行政用印佇列自動排入

研究結論（電子簽章法 2024 修法）：效力三階 = 電子簽名 < 電子簽章 < **數位簽章（推定本人簽署、舉證責任反轉）**。

| 場景 | v1 做法 | 演進方向 |
|------|---------|----------|
| **內部簽核用印**（低法律風險） | 自建簽核引擎（上述）、NuEIP 為可選 adapter | 簽核引擎開放給所有卡帶複用 |
| **對客戶簽署**（高法律風險） | 沿用現行 email 回簽基準 + 回簽檔案自動落檔 | 外接點點簽 DottedSign API（政府雲市集背書、OTP + 數位憑證雙軌）— 憑證引擎不自建 |

所有簽署動作強制記錄稽核軌跡（誰 / 何時 / 憑證）— 這是強制欄位、不是加值功能。

## 7.5 Agentic 差異化 — 超越底稿的五個代理能力

Allen 底稿的世界觀是「看板 + 通知」：系統把狀態擺出來、提醒人去做。**BizPeak Flow 的世界觀是「代理 + 拍板」：系統把事情做到只剩最後一個決定、人只負責拍板。**這是與現有體驗的代差、也是「AI 原生公司的內部系統」該有的樣子。每個代理動作都遵守信任邊界：AI 起草與建議 → 人審 → 信任橋接寫入（憲法第三、四條）。

| 代理能力 | 現況（人工） | BizPeak Flow（代理） |
|----------|--------------|----------------------|
| **回簽識別** | 有人看到 sales@ 的信、轉給財務 | AI 判讀進信是哪張合約的回簽、抽取預計付款日與金額、自動落檔並通知 — 人只在抽取有疑義時介入 |
| **收款對帳** | 財務看銀行明細、逐筆對、月底結帳 | 銀行入帳資料（明細匯入 / 通知信）→ AI 比對款項排程 → 產生「建議打勾」清單、財務一鍵確認 |
| **催收代辦** | 業務想起來才追、信自己寫 | 代理依客戶分群與歷史行為起草催收信、附上下文（合約、逾期天數、過往互動）、業務按一下送出 |
| **風險預警** | 沒有 — 逾期才知道 | 依客戶歷史付款行為預測逾期風險、開票前就標記高風險客戶、建議調整付款條件 |
| **主動彙報** | 沒有 — 要人去查 | 代理在 Teams 主動說話：「今天 2 張合約卡在用印超過 SLA、1 筆款項預計逾期」— 通知是可操作的（卡片上直接核准 / 升級）、不是死訊息 |

**驗收判準（宣告式）**：財務的月底對帳時間從半天降到一小時內；業務從「自己記得要追」變成「收到已起草好的信、按一下」。

**與訪談的關係**：這五項是提案的「超出期待」基準線 — 訪談不是問「要不要」、是問「順序與細節」。

## 8. 三階段體驗與驗收判準（v0.3 定義修正、Nelsen 2026-07-02）

| 階段 | 定義 | 驗收判準（宣告式） |
|------|------|--------------------|
| **1.0 圖形介面** | 完整的合約管理系統：合約列表、生命週期追蹤、收款佇列、逾期看板（三入口、Notion 資料層、接報價單系統輸出） | 一張真實合約從 C0 走到 C6、全程狀態可查、通知真的送達、財務真的用它打勾收款 |
| **1.5 介面＋聊天框** | **在 1.0 的既有介面上長出聊天框** — 說一句話、介面自己動：導航到對應功能、開好表單、給連結點過去確認（類 Gamma / Notion 內建 Agent 的模式） | 聊天框說一句話、介面完成導航與草稿、人審後入庫 |
| **2.0 AI 應用 × MCP** | **使用者在自己慣用的 AI 聊天應用**（Claude / GPT 等）、透過 Funraise MCP 把個人情境（信箱、桌面檔案）與公司合約系統接進同一場對話 — 回簽信直接歸檔、報價試算直接更新、必要時把介面元件帶進聊天 | 在外部 AI 助理完成一次跨系統操作（信箱回簽 → 合約歸檔）、全程不開我們的介面 |

**能力層先行（憲法第二條）**：1.0 的每個按鈕背後都是一個可程式呼叫的能力 — 1.5 / 2.0 不重寫、只換皮。1.0 → 1.5 的對比刻意極小化：同一個畫面、多一個聊天框 — 讓「AI 化」的體感一眼可見。

## 9. 成功指標（建議、待 Nelsen 拍板）

1. **收款追蹤覆蓋率 100%** — 每張發票在系統有狀態、無帳外追蹤
2. **逾期發現延遲**：月底結帳（現況、最長 30 天）→ **D+1 自動浮現**
3. **回簽 → 財務知道**：人工轉達（不定時）→ **5 分鐘內自動通知**
4. **續約提醒觸發率 100%**（到期前 60 天無一漏網）
5. **新人上手 ≤ 1 小時**（讀入口 + SOP 即可獨立跑完整流程）
6. **卡帶接入數**（主幹道效應的長期指標）：6 個月內 ≥ 2 個先鋒計畫卡帶通過合規接入
